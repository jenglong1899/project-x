# async memory manager and keep last n msg

keep-last-n有个问题：如果keep-last-n对应的内容正好很大，那么等会触发检查的时候，judge就会认为当前上下文中有50%以上的内容都是不重要的，然后又触发重置

可能的解决办法1：弄一个保护期？重置后，跳过3个阈值

可能的解决办法2：

要弄keep-last-n的原因就是：做摘要和决定重置都是异步的
```
（一些消息）
<wake-memory-manager-flag>
（worker继续运行）
（此时summary完成摘要任务，judge决定重置上下文。现在这个点，到上一个flag，这中间的内容是没有被摘要的）
```
所以得keep last n msg，来保证这部分没做摘要的不被丢掉

那么可以：此时停下worker，
如果这中间的msg不是很大，那么就keep这中间的msg，放到新的上下文中
如果很大的话，就要做摘要。

那如果用户刚发了一大堆消息，内容很大，然后紧跟不久，judge决定重置上下文，那么你这个摘要可能就麻烦了，因为这些用户发的这东西根本没有没法被摘要，可能真的很有价值。
这种情况出现的可能比较少，也可以不去考虑。

那你可以说 keep last n 那个抖动出现的可能也比较小。这么说确实是对的，但是keep last n感觉确实是可以优化掉的。

或者不是keep last n，"现在这个点，到上一个flag，这中间的内容是没有被摘要的" 那就把这段全都保留下来就好了。这个就非常简洁了。就算这段内容非常大，那我们也可以到下一次总结。

# worker和mm的todo不同步问题

Memory Manager 做摘要的时候，极大概率是要记录以前做了什么事情，还有接下来要做什么事情的。
Worker 自己也维护了一个 todo。
这就重合了。

比如到了一个 Memory Manager 的触发点，开始了一个摘要任务，然后开始了一个重置决策任务。然后worker继续运行，然后这个时候worker完成了一个东西，然后划掉了一个 to do， 但是做摘要那个 memory manager 他还没有这方面的上下文，他会认为那个还没完成。

所以问题解决方案其实是，当决定重置后，worker要暂停，然后这时候再次触发一次 summarizer。

## 先开始judge，然后再summary？

上面的做法的问题可能是：summary刚做完没多久，然后judge决定重置，然后summary又要被唤醒一次，这就好像有点浪费
要不要先唤起judge（不留flag），然后等5秒，然后唤起summary（留flag）
把这个 flag 改名一下，得改成 WAKE_MM_SUMMARY_FLAG ，这样才符合实际。
不过可能也没有必要先做这个了，这个功能有最好，但没有也可以？

## 方法2
决定重置后，worker要暂停，但是不要再触发一次summarizer，而是把上次summary的位置和当前这个位置的内容保留下来，在下一次重置后加载。
judge判断重置应该不会需要很久，可能5-10秒左右，这个时间段内，worker的上下文大概率不会增长太多，这个时候触发summary会有点浪费。

# 再压缩？

我平常问 ChatGPT 都是很杂的问题。问一下代码，然后问一下历史，然后问一下游戏。

用户在使用我们这个产品的时候，产品是会给AI挂个主要记忆文档，然后用户问了一个和主要记忆里面都无关的问题。那这么时候，AI是不是可以决定下一轮自己不要去加载这个完整的主要记忆文档了，就是你加载那些主要记忆文档的那些标题就行了，或者说单独另外一个更加摘要的文件就行了。

这已经进行过一轮运算了，下一轮就是有缓存的了。你得权衡一下你这个最后的摘要和缓存，那个价格更低。我猜测应该是1:1吧。
