# async memory manager and keep last n msg

keep-last-n会有个“抖动问题”，如果keep-last-n对应的内容正好很大，那么等会触发检查的时候，judge就会认为当前上下文中有50%以上的内容都是不重要的，然后又触发重置

可能的解决办法1：弄一个保护期？重置后，跳过3个阈值

可能的解决办法2：

要弄keep-last-n的原因就是：做摘要和决定重置都是异步的
```
（一些消息）
<wake-memory-manager-flag>
（worker继续运行）
（此时summary完成摘要任务，judge决定重置上下文。现在这个点，到上一个flag，这中间的内容是没有被摘要的）
```
所以得keep last n msg，来保证这部分没做摘要的不被丢掉

那么可以：此时停下worker，
如果这中间的msg不是很大，那么就keep这中间的msg，放到新的上下文中
如果很大的话，就要做摘要。

那如果用户刚发了一大堆消息，内容很大，然后紧跟不久，judge决定重置上下文，那么你这个摘要可能就麻烦了，因为这些用户发的这东西根本没有没法被摘要，可能真的很有价值。
这种情况出现的可能比较少，也可以不去考虑。

那你可以说 keep last n 那个抖动出现的可能也比较小。这么说确实是对的，但是keep last n感觉确实是可以优化掉的。

或者不是keep last n，"现在这个点，到上一个flag，这中间的内容是没有被摘要的" 那就把这段全都保留下来就好了。这个就非常简洁了。就算这段内容非常大，那我们也可以到下一次总结。

# 文件内容引用语法
制定一种md语法，在记忆文档中用这个语法，系统到时候build user level instruction的时候就会自动加载对应的文档的内容。

这个语法支持行号，可以引用对应行号范围的内容，而不是整个内容

这样一次性就可以加载完，节省token

如果上下文中有很多文件的内容都是需要的，那么这个就很有利于记忆系统，你打算重置的时候，引用这些文件就行了。

# fork_memory_manager_tool

用户可以主动唤起memory manager，让它去重置。
我们希望 Worker 和 Memory Manager 尽可能分离，就是 Worker 尽量不要涉及去跟记忆相关的事情。
所以是用户主动去跟 Memory Manager 谈，而不是说给 Worker 一个工具让他去唤醒 Memory Manager。

可以在不暴露底层实现的情况下实现这个功能。在system prompt里写了会有专门的memory agent，如果用户有记忆方面的指令，唤起它就行了（平时系统会自动唤起memory agent），这就是一种分离。

system prompt:
```
系统会定期自动fork一个memory manager出来让它工作，但如果用户主动要求worker去做记忆相关的操作，worker也可以调用工具 fork_memory_manager 来主动唤起memory manager来处理记忆
```

当worker主动调用了 fork_memory_manager 后，系统定期自动fork的memory manager的频率也要随之调整。比如前脚刚fork了，然后后脚系统又自动唤起了一次。

这功能不是最先需要的功能，先放着。

# 再压缩？

我平常问 ChatGPT 都是很杂的问题。问一下代码，然后问一下历史，然后问一下游戏。

用户在使用我们这个产品的时候，产品是会给AI挂个主要记忆文档，然后用户问了一个和主要记忆里面都无关的问题。那这么时候，AI是不是可以决定下一轮自己不要去加载这个完整的主要记忆文档了，就是你加载那些主要记忆文档的那些标题就行了，或者说单独另外一个更加摘要的文件就行了。

这已经进行过一轮运算了，下一轮就是有缓存的了。你得权衡一下你这个最后的摘要和缓存，那个价格更低。我猜测应该是1:1吧。

除非这个功能可以省很多钱，否则就不做这个了，没有必要为此徒增复杂性。
