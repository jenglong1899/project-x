注意，这个方案仍然是fork，也就是沿用之前的上下文，不新开上下文

做摘要、决定要不要重置上下文是两个比较大的任务，因此需要做拆分。

而且做摘要和重置上下文是两个独立的任务。
你做摘要只管摘要就行了，不需要管是不是要到重置上下文的时候了（不需要判断当前上下文是否有50%以上是没用的）
你做重置上下文，也不需要管现在做了什么摘要，你判断当前上下文是否有50%以上的内容是没用的，你做这件事根本就不需要知道做了啥摘要。

上下文每增长3%（按模型上下文窗口计算,(current_tokens - last_checkpoint_tokens) / context_limit >= 3%），这种就会自动唤起一次摘要任务，一次判断重置任务。

重置上下文的标准：
- 如果当前上下文中有50%以上的内容都是对当前任务不重要的，那我们就可以考虑重置。一个例子是上下文中有大量的中间过程，而我们只需要最后的结果
  - 这里的50%是按token来算，我到时候会给一个工具让memory manager调用，这个工具会显示当前上下文中：
    - user msg 占据多少百分比的上下文窗口
    - AI msg 占据多少百分比，且其内部还会再细分：
      - reasoning占据多少
      - content占据多少
      - tool call占据多少
    - tool result 占据多少百分比，且其内部还会再细分：
      - 各个工具的tool result的占比（大于一定比例才显示，小于一定比例的，通通归为“其他工具”）
- 我们需要判断这个任务还有多少轮完成吗？如果当前上下文全都是对完成那些任务有用的信息，那我们不管预估还多少轮才能完成当前任务，我们都要留着。
- 我们需要在空闲的时候自动摘要+重置吗？不需要，仍然是以“当前上下文是否有50%的内容都是对当前任务不重要的”为判断标准，没准用户空闲了一阵子，回来还是继续那个任务呢？


在memory manager做出reset的决定后，系统要等所有的摘要做完。

做摘要，是可以异步的。因为worker也不需要知道memory manager做了什么摘要

做摘要的话，我们可能需要在上下文中留下一些标识，标识出上一次做摘要任务是在什么位置，提醒memory manager：“通常来说，你只需要关注你上一次做摘要的地方到现在这个地方的内容，在这之前的内容已经被之前的你做过摘要了”
```
一些内容（user msg, ai tool call, tool result）
系统调用memory manager后，自动留一条user msg: <memory_manager_awaken_flag>
一些内容（user msg, ai tool call, tool result）
系统调用memory manager后，自动留一条user msg: <memory_manager_awaken_flag>
一些内容（user msg, ai tool call, tool result）
```

其实我觉得这个也还行吧。我们也不用完全原封不动的说出来，就是我们说前几个字就可以了。好吧，不过有重复怎么办？重复的概率实在太低了吧。不用管这个。

第三种方法，系统会自动给所有msg加上一个全局唯一单调递增的数字id，比如用户输入了一个消息，系统会把它变成：`{id} {用户说的话}`，AI的回复，系统在把它加入到上下文之前，会把它的回复也加上一个数字id。对于tool result，我们会在其json中加一个id字段

理论上做摘要不需要很久，系统在自动唤起下一次摘要之前，上一次摘要应该做完了，但是以防万一，如果上一次还没做完，系统就应该延迟到下一次。
